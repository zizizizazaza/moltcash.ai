# AI 驱动项目管理体系

> 本 skill 总结了一套经过实战验证的 AI-人类协作开发模式。适用于任何全栈 Web 项目。

---

## 1. 核心理念

| 原则 | 说明 |
|------|------|
| **文档即真相** | 所有决策、进度、架构都写在 `docs/` 中，AI 和人类共读一份文档 |
| **渐进式细化** | 先有模块总览，再逐步展开功能明细和待办 |
| **人机分工清晰** | 用 ✅/⚠️/❌ + 👤/AI 标记，一目了然谁做什么 |
| **单一信息源** | TODO.md 是唯一进度追踪文件，不散落在多处 |

---

## 2. 文档体系（`docs/` 目录）

每个项目应包含以下文档：

```
docs/
├── requirement.md          # PRD — 唯一权威需求文档
├── ARCHITECTURE.md         # 技术架构 + 设计决策 + 目录结构
├── API.md                  # 完整接口文档（请求/响应示例）
├── DEVELOPMENT.md          # 开发指南（环境搭建/命令/联调/FAQ）
├── TODO.md                 # 模块进度追踪（核心！）
└── TEST.md                 # 测试用例（按用户路径组织）
```

### 2.1 各文档职责

| 文档 | 谁写 | 何时更新 | 核心内容 |
|------|------|---------|---------|
| `requirement.md` | 👤 人类 | 需求变更时 | 产品定义、用户故事、功能范围 |
| `ARCHITECTURE.md` | 🤖 AI | 架构变更时 | 技术栈、分层、目录结构、设计决策表 |
| `API.md` | 🤖 AI | 每次改 API 后 | 端点、请求/响应 JSON、认证方式 |
| `DEVELOPMENT.md` | 🤖 AI | 环境/流程变化时 | 首次启动、常用命令、联调说明、FAQ |
| `TODO.md` | 🤖 AI | **每次功能完成后** | 模块总览、功能明细、上线待办 |
| `TEST.md` | 🤖 AI | 功能完成后 | 按用户路径的测试矩阵 |

### 2.2 文档自动同步

如果使用 Gemini UI 查看 artifact，需要通过 `write_to_file`（`IsArtifact=true`）同步。
参考 `.agents/workflows/sync-docs.md` 了解同步规则。

## 3. TODO.md 模板（核心）

TODO.md 是整个项目管理的心脏。以下是结构模板：

```markdown
# 🚀 项目名 — 项目进度 & 上线追踪

> **最后更新**：YYYY-MM-DD · **整体代码进度**：~XX% · **上线就绪**：~XX%

---

## 📊 上线路径总览

用 Mermaid graph 展示模块间依赖关系。

## 📋 模块上线总览

用表格一行一模块，列出：子功能数、✅/⚠️/❌ 各多少、就绪度百分比、主要阻塞。

## 🔍 模块功能明细

每个模块一个 ### 标题，内部包含：
1. **功能表格**：子功能 | UI | 后端 | 链上(可选) | 说明
2. **上线待办**：checkbox 列表，每项标注 👤 或 AI

## 📦 开发阶段完成情况

按 Phase 分组，展示历史开发进度。

## 👤 你的工作清单

列出所有需要人类操作的事项（域名/服务器/API Key/合规等）。

## ⏭️ 下一步行动

Gantt 图展示 MVP 上线路径。
```

### 3.1 状态标记规范

| 标记 | 含义 | 用于 |
|------|------|------|
| ✅ | 真实可用、已完成 | 功能状态 |
| ⚠️ | 部分真实、有限制 | 功能状态 |
| ❌ | 假数据或未实现 | 功能状态 |
| 🚫 | 功能完全缺失 | 功能状态 |
| 🔴 | 低就绪 (0-30%) | 模块就绪度 |
| 🟡 | 中就绪 (30-75%) | 模块就绪度 |
| 🟢 | 高就绪 (75-100%) | 模块就绪度 |
| 👤 | 需要人类操作 | 上线待办 |
| AI | AI 可完成 | 上线待办 |
| [x] | 已完成待办 | checkbox |
| [ ] | 未完成待办 | checkbox |

### 3.2 更新规则

1. **每完成一个功能**，更新对应模块的功能表格和上线待办
2. **每次 session 结束前**，如有进度变化，更新模块总览表的 ✅/⚠️/❌ 计数和就绪度
3. **整体进度百分比** = 所有模块就绪度的加权平均

## 4. 人机协作流程

### 4.1 功能开发流程

```
1. 👤 提出需求（自然语言描述）
2. 🤖 调研现有代码 → 提出方案
3. 👤 审核方案 → 批准/调整
4. 🤖 实现功能（前端 + 后端）
5. 🤖 更新 TODO.md（标记完成 + 调整就绪度）
6. 👤 验证功能（浏览器测试）
7. 🤖 修复反馈问题
```

### 4.2 调试流程

```
1. 👤 描述问题（文字 + 截图）
2. 🤖 查看相关代码 → 定位原因
3. 🤖 修复 → 说明改了什么
4. 👤 验证修复
```

### 4.3 进度汇报

- 每次显著功能完成后，更新 TODO.md
- 用户可以随时要求 "更新 todo" 来同步进度
- 用 artifact 同步 TODO.md 供 Gemini UI 渲染
- Mermaid 图表提供可视化总览

## 5. ARCHITECTURE.md 模板

```markdown
# 项目名 — 架构文档

## 1. 项目定位
一句话说明项目做什么。

## 2. 技术总览
ASCII 图展示前端/后端/数据库/第三方服务的关系。

## 3. 目录结构
完整的项目目录树，每个文件/目录加注释。

## 4. 前端架构
路由方案、状态管理、API 客户端、样式方案。

## 5. 后端架构
分层结构、认证机制、AI 服务、WebSocket、数据库。

## 6. 关键设计决策
表格列出：决策点 | 选择 | 原因。

## 7. 当前开发状态
已完成和待做的概览（指向 TODO.md）。

## 8. 端口 & 环境变量
表格列出所有端口和 .env 变量。
```

## 6. API.md 模板

```markdown
# 项目名 — API 接口文档

> 基础 URL | 认证方式

## 认证标记说明
🔓 无需认证 | 🔒 需要 JWT | 🔑 可选认证

## 按模块分组
每个端点包含：
- HTTP Method + Path + 认证标记
- Request Body (JSON 示例)
- Response (JSON 示例，含状态码)
- 自动计算逻辑说明

## 错误响应格式
统一的错误 JSON 结构。
```

## 7. DEVELOPMENT.md 模板

```markdown
# 项目名 — 开发指南

## 1. 环境要求
表格：工具 | 版本 | 说明

## 2. 首次启动
前端 + 后端的启动命令。

## 3. 常用命令

## 4. 前后端联调
代理配置 + 登录流程 + 手动 API 测试。

## 5. 数据库操作
查看/重置/修改 Schema/种子数据。

## 6. 添加新功能指南
新增 API/页面/数据表的步骤清单。

## 7. 项目约定
代码风格/文件命名/Git 忽略。

## 8. 常见问题
端口占用/Prisma 类型过期/AI 不响应 等。
```


## 9. 新项目 Quick Start

在新项目中应用本体系的步骤：

1. **创建 `docs/` 目录**
2. **从 `requirement.md` 开始** — 写清楚产品要做什么
3. **拆分模块** — 在 TODO.md 中列出所有模块（M1, M2, ...）
4. **每个模块列功能** — 用 ✅/⚠️/❌ 标记当前状态
5. **标注人机分工** — 哪些 AI 能做，哪些需要 👤
6. **开始开发** — 按模块优先级逐个实现
7. **持续更新 TODO.md** — 每完成一个功能就更新
8. **需要时补充其他文档** — ARCHITECTURE/API/DEVELOPMENT/TEST

## 10. 关键经验

### ✅ 有效的做法
- TODO.md 用 **表格** 而非纯文字，一目了然
- 用 **Mermaid** 画模块依赖图和 Gantt 图
- 功能表格分 **UI / 后端 / 链上** 列，精确标记
- 上线待办用 **checkbox + 角色标记**
- 每个模块给一个 **就绪度百分比**

### ❌ 避免的做法
- 不要把 TODO 散落在代码注释里
- 不要让 TODO.md 超过 500 行（太长则拆分模块文档）
- 不要省略 "说明" 列 — AI 下次 session 需要上下文
- 不要只标记 ✅/❌ 不写原因 — ⚠️ + 说明更有用
